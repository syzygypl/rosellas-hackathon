# Add New Application

## Overview

Use this skill to add an application without drifting from the repository's Nx, Docker, deployment, and versioning conventions.

## Required Context

1. Read `skills/start-development-session/SKILL.md`.
2. Read `skills/start-development-session/references/project-context.md`.
3. Read `docs/versioning/README.md` before implementing version endpoints, UI badges, Swagger metadata, or deploy build metadata.
4. Run or inspect `git status --short` before editing and preserve unrelated user changes.

## Placement

- Put example applications under `apps/examples/<app-name>/`.
- Keep a single root `package.json` and `package-lock.json`; do not add per-app package files unless the user explicitly changes the workspace model.
- Give every app an Nx `project.json` with explicit build outputs.
- Add or update root scripts only when they improve common workflows.

## Backend Checklist

For NestJS backends:

- Keep the public API under `/api`.
- Register controllers and modules through the app module.
- Document DTOs and endpoints with `@nestjs/swagger`.
- Add Swagger UI if the backend is public or user-facing.
- Add the version endpoint and Swagger versioning from `docs/versioning/README.md`.
- Keep Cloud Run runtime configuration in environment variables.

## Frontend Checklist

For Angular frontends:

- Keep API access centralized in Angular services.
- Keep environment values in `src/environments/`.
- Display frontend version metadata and paired backend metadata when the app talks to a backend.
- Use compact UI for metadata; do not turn version display into primary page content.
- Keep production API URLs generated by CI when they depend on deployed Cloud Run URLs.

## Build And Docker

- Build application artifacts through Nx before Docker packaging.
- Dockerfiles should copy prebuilt artifacts instead of running Nx builds inside Docker.
- Ensure Nx `outputs` match the Dockerfile `COPY` paths.
- Keep Docker images service-specific.
- Do not commit generated `dist/`, `.angular/`, or local credential files.

## Workflows And Docs

- Update `.github/workflows/` when the app has an independent deploy path.
- Name each deploy workflow after the Cloud Run service it deploys: `.github/workflows/<service-name>.yml` with `name: <service-name>`.
- Include app paths, root package files, and Nx config in workflow path filters.
- Update `README.md` for user-facing commands or runtime behavior.
- Update `skills/start-development-session/references/project-context.md` for architecture, commands, deploy behavior, endpoints, or important paths.
- Update `docs/versioning/README.md` if the versioning contract changes.

## Verification

Use the narrowest useful verification:

- Backend app: `npm run build:backend` or the app-specific Nx build.
- Frontend app: `npm run build:frontend` or the app-specific Nx build.
- Cross-service/root config: `npm run build`.
- Docs-only changes: review rendered Markdown links and referenced paths.

Do not run live GitHub or GCP deployment checks unless the user asks for live validation.
